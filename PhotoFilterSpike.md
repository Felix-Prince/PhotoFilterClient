## 为什么要做 Spike

三份主文档里有若干假设是**没有实测依据的猜测**——猜对了项目顺利，猜错了在阶段一/三才发现就要回头改架构。Spike 阶段花 4-6 小时做实验，把"假设"变成"已知"。

每个 Spike 都遵循相同结构：

1. **要验证的假设** — 明确写出来要证伪/证实什么
2. **实验方法** — 怎么跑、用什么素材、记录什么数据
3. **决策门** — 数据达到什么程度算通过、不通过怎么调整架构
4. **回填位置** — 实验完成后要回头修订哪份文档的哪一节

## Spike 1：模型分流可行性（1-2h）

### 要验证的假设

> gemma4:e4b 对"倾向保留 / 倾向淘汰 / 不确定"三分类的分流是有意义的——大部分照片能被明确归到前两类，只有少数边界照片落到"不确定"，从而省下 InternVL3.5 的调用量。

如果"不确定"占比过高，轻量预判层就是冗余的，应改为单层架构（全量直接走 InternVL3.5）。

### 实验方法

1. 准备**场景多样子集** 100 张
2. 设计 prompt（先用最朴素版本，不要花时间调优——这一步是验证模型能力，不是验证 prompt）：

   ```
   你是一个摄影筛选助手。请快速判断这张照片：
   - 倾向保留 (keep)：直觉上有亮点，值得留下
   - 倾向淘汰 (drop)：明显平庸/有硬伤
   - 不确定 (unsure)：拿不准，需要细看

   只输出一个 JSON：{"verdict": "keep" | "drop" | "unsure", "reason": "<一句话>"}
   ```

3. 用 gemma4:e4b 跑 100 张，记录每张的 verdict
4. 用户**人工**对同 100 张做相同三分类判断（5-10 分钟）
5. 比对模型分类与人工分类

### 记录字段

| 字段 | 说明 |
|------|------|
| 照片 ID | 文件名 |
| 模型 verdict | keep / drop / unsure |
| 模型 reason | 模型给的理由 |
| 单张耗时 | 毫秒 |
| 人工 verdict | 用户实际判断 |
| 是否一致 | TRUE / FALSE |

### 决策门

| 指标 | 阈值 | 含义 |
|------|------|------|
| 模型 unsure 占比 | < 40% | 大部分照片能被分流，轻量层有价值 |
| 模型与人工一致率（不计 unsure） | ≥ 75% | 模型判断方向基本对 |
| 单张耗时 | < 3s | 800 张全量预判可控制在 ~40 分钟内 |

**通过** → 保留双层架构，轻量预判 + 边界深度评估。

**不通过**（unsure 占比 > 40% 或一致率 < 60%）→ 改为单层架构，全量直接走 InternVL3.5；轻量层作为"无 GPU 时的降级方案"保留但不默认启用。

### 回填位置

- [PhotoFilterDesign.md §12.2](PhotoFilterDesign.md) "实际执行策略"——可能改成单层
- [PhotoFilterApplyPath.md 阶段三 3b](PhotoFilterApplyPath.md) "轻量预判 + 深度评估流程"——可能简化

### 实测结果（已完成，2026-06-16）

```
执行日期: 2026-06-16
照片量: 63（E:\摄影集\2026-03-29-下沙江滨樱花）
实测模型: gemma4:e4b
模型 keep 占比: 100.0%
模型 drop 占比: 0.0%
模型 unsure 占比: 0.0%
平均单张耗时: 6.58s
最长单张耗时: 11.23s（首张含模型加载）

决策结论: 改单层
理由: 虽然 unsure=0%，但 63 张同场景照片全部输出 keep，零区分力；平均耗时 6.58s 也超过轻量预判阈值。轻量预判层不可用，默认全量走 InternVL3.5。
```

## Spike 2：评分稳定性测试（1-2h）

### 要验证的假设

> InternVL3.5 + temperature=0 + 固定 seed 下，同一张照片连续评估 3 次的**结论**（保留/淘汰/不确定）一致率 ≥ 95%，**分数**抖动 ≤ ±5 分。

如果不达标，[PhotoFilterDesign.md §4.8](PhotoFilterDesign.md) 设计的"评分稳定性约束"就是无效的，需要改方案（更强的离散化 / 更激进的 self-consistency 投票次数 / 换模型）。

### 实验方法

1. 准备**稳定性子集** 30 张（含明显好 10 张 + 明显差 10 张 + 边界 10 张）
2. 设计审美评估 prompt（要求输出整数 0-100 + 结论 keep/drop/unsure）：

   ```
   评估这张照片的审美价值，按四个维度各给 0-100 整数分（不要小数）：
   - 视觉吸引力 (visual)
   - 情感共鸣 (emotion)
   - 叙事性 (narrative)
   - 稀缺性 (rarity)

   并给出最终结论：keep / drop / unsure

   只输出 JSON：
   {"visual": 0-100, "emotion": 0-100, "narrative": 0-100, "rarity": 0-100, "verdict": "keep|drop|unsure"}
   ```

3. 配置 Ollama 调用：`temperature=0`、`seed=42`、`top_k=1`
4. 每张照片连续调用 **3 次**（共 90 次），记录每次的完整 JSON
5. 计算结论一致率与各维度分数抖动

### 记录字段

| 字段 | 说明 |
|------|------|
| 照片 ID | 文件名 |
| 第 1 次 JSON / 第 2 次 / 第 3 次 | 完整输出 |
| 三次 verdict 是否一致 | TRUE / FALSE |
| 各维度三次分数极差（max-min） | 数字 |
| 单张三次平均耗时 | 秒 |

### 决策门

| 指标 | 阈值 |
|------|------|
| **结论一致率**（3 次完全一致的照片占比） | ≥ 95% |
| **各维度分数抖动**（三次极差的中位数） | ≤ 5 |
| **JSON 解析成功率** | 100% |
| 单次评估耗时（参考） | 记录但不卡阈值 |

**通过** → 设计文档 §4.8 的稳定性约束有效，照常执行。

**不通过**（一致率 < 95%）→ 启动备用方案，按代价从小到大尝试：

1. self-consistency 投票次数从 3 增加到 5
2. 加严 prompt 的输出限制（如要求"分数必须是 5 的倍数"，强制对齐到 5 档）
3. 换模型（InternVL3.5-2B 反而可能更稳定，因为输出空间小）

### 回填位置

- [PhotoFilterDesign.md §4.8](PhotoFilterDesign.md) "评分稳定性约束"——可能加严
- [PhotoFilterApplyPath.md 阶段三验收](PhotoFilterApplyPath.md)——把测得的实际阈值写进去作为基线

### 实测结果（已完成，2026-06-16）

```
执行日期: 2026-06-16
照片量: 30 × 3 次 = 90 次调用
实测模型: bInternlVL35:8b（InternVL3.5）
结论一致率: 100.0%
各维度分数抖动中位数: 0 / 0 / 0 / 0  (visual/emotion/narrative/rarity)
JSON 解析成功率: 100.0%
平均单次耗时: 2.76s

决策结论: 稳定性达标
理由: temp=0 + seed=42 下完全稳定，设计文档 §4.8 的稳定性约束有效。
具体调整: 保留 5 档展示；后续 prompt 调优重点转为提高分数区分度（当前分数集中在 85/90/95）。
```

## Spike 3：缩略图加载与 Tauri IPC 性能（1-2h）

### 要验证的假设

> Tauri v2 + Vue 3 + 虚拟滚动方案下，800 张照片的缩略图首屏加载 < 5s、滚动 FPS ≥ 30，且 IPC 传图不会阻塞主线程。

如果不达标，要么换虚拟滚动库、要么改 IPC 协议（如改为前端直读本地文件 URL 而非 base64 IPC）。这种问题阶段一才发现的话整个 UI 层要返工。

### 实验方法

1. 搭最小 Tauri v2 + Vue 3 项目（不实现完整功能，只跑缩略图加载链路）
2. Rust 后端实现：
   - 扫描指定文件夹的所有 JPG/RAW
   - 为每张照片生成 200x200 缩略图（缓存到临时目录）
   - 提供 IPC 接口 `get_thumbnail(path) → bytes` 或 `get_thumbnail_url(path) → file:// URL`
3. 前端实现：
   - 使用 vue-virtual-scroller 渲染卡片网格
   - 每张卡片懒加载缩略图
4. 用 800 张测试集跑：
   - 测**首屏加载时间**：从打开窗口到首屏卡片可见
   - 测**滚动 FPS**：用 Chrome DevTools Performance / Tauri devtools 测快速滚动时的帧率
   - 测**内存占用**：滚动到底部后的进程内存
   - 测**IPC vs file:// URL 两种方案的差异**：哪个更快、谁阻塞主线程

### 记录字段

| 测试项 | base64 IPC 方案 | file:// URL 方案 |
|--------|----------------|------------------|
| 首屏时间 | __s | __s |
| 滚动 FPS（中位数）| __ | __ |
| 滚动 FPS（最低值）| __ | __ |
| 800 张全部滚动到底后内存 | __MB | __MB |
| 主线程是否卡顿 | TRUE / FALSE | TRUE / FALSE |
| 首次缩略图生成总耗时 | __s | __s |
| 缩略图缓存命中后的二次加载 | __s | __s |

### 决策门

| 指标 | 阈值 |
|------|------|
| 首屏加载 | < 5s |
| 滚动 FPS（中位数）| ≥ 30 |
| 滚动 FPS（最低值）| ≥ 20（不能掉到肉眼可感知卡顿）|
| 主线程不阻塞 | 必须 |
| 内存占用（800 张）| < 1GB |

**通过** → 选择性能更好的方案（base64 IPC vs file:// URL），写入架构文档。

**不通过**：

- 首屏 > 5s → 缩略图分批加载（前 50 张优先），其余懒加载
- 滚动卡顿 → 换虚拟滚动库（如 vue-virtual-scroll-list），或减小卡片尺寸
- 内存 > 1GB → 滚出可视区的卡片释放图片资源
- 极端情况：Tauri v2 表现明显劣于预期 → 启用切换预案，回到 Electron

### 顺手测一下（如果时间允许）

- **跨磁盘性能**：把 800 张照片放到外置硬盘/NAS，重测一遍。验证 [PhotoFilterDesign.md §9.3](PhotoFilterDesign.md) "跨磁盘明示"的必要性
- **文件移动事务化**：实现一个最小版本（移动 5 个 JPG+RAW 配对到子文件夹），故意中断（Ctrl+C 或拔 U 盘）一次，看回滚是否可靠
- **SHA-256 全文件耗时**：800 张 RAW 全部算一遍 SHA-256，记录总耗时——决定第三轮 Info 10 的哈希策略

### 回填位置

- [PhotoFilterArchitectureUI.md §1.1](PhotoFilterArchitectureUI.md) Tauri v2 切换预案——根据实测结果决定是否触发
- [PhotoFilterArchitectureUI.md §6](PhotoFilterArchitectureUI.md) 虚拟滚动方案——写入选定的库与配置
- [PhotoFilterArchitectureUI.md §8.5](PhotoFilterArchitectureUI.md) 哈希策略——按 SHA-256 实测决定是否需要混合指纹

### 实测结果（已回填）

```
执行日期: 2026-06-16
照片量: 130 张（JPG + RAF 配对）
最终采用方案: 逐张 base64 IPC + 客户端 3 路并发
首屏时间: 缓存命中后秒级，未精确测（非瓶颈）
滚动: 流畅，未卡顿
内存占用: 130 张稳定，未崩溃
跨磁盘对比: 工作目录在 E: 盘，rename 同卷秒级，无降级问题
SHA-256 全量耗时: 未单独测（采用简单 hash，非 SHA-256）

文件移动事务化中断测试: 通过（配对原子性 + 回滚在 bug 修复后验证可靠）

决策结论: Tauri v2 通过
具体调整:
  1. 同步命令会冻结 UI → 改为 async + spawn_blocking
  2. 大批量 base64 单次 IPC 必崩 → 改逐张 + 并发控制
  3. convertFileSrc 在 Windows + Tauri v2 配置复杂 → 暂用 base64
  4. 配对收集大小写双重 bug → 改为遍历目录按 stem 收集
```

> **阶段三实测发现（非 Spike 但相关）**：阶段一实现中暴露了几个工程约束，已回填到 ArchUI §1.2——Tauri 命令必须 async、缩略图逐张 + 并发、不使用 Mutex 全局锁（会死锁）。

## 备注

- 三个 Spike **可以并行**：Spike 1/2 是模型层的脚本测试，与 Spike 3 的 Tauri 骨架开发互不阻塞
