## 一、技术选型

### 1.1 桌面应用框架：Tauri v2

| 维度 | 选择 | 说明 |
|------|------|------|
| 框架 | Tauri **v2.x** | 包体小(~10MB)、性能好、Rust 后端文件操作自然；v2 长期维护 |
| 后端语言 | Rust | 直接调 OS API，文件移动/扫描/XMP 读写 |
| 前端框架 | Vue 3 | 上手简单、中文生态好、Tauri 官方支持 |
| 前端构建 | Vite | 快，Tauri + Vue 模板默认集成 |

**核心 Tauri 插件**（v2 命名）：

| 插件 | 用途 |
|------|------|
| `tauri-plugin-fs` | 文件系统读写、扫描、移动 |
| `tauri-plugin-dialog` | 文件夹选择对话框 |
| `tauri-plugin-shell` | 双击照片调用系统默认查看器 |
| `tauri-plugin-sql` | SQLite 决策历史与偏好存储 |
| `tauri-plugin-store` | 偏好画像与配置的轻量 KV 存储 |

> **版本一致性**：所有 Tauri 插件必须与 Tauri 核心保持同一大版本（v2）。spike 阶段如发现某插件 v2 不稳定，优先寻找替代而非回退到 v1——v1/v2 的 IPC 协议与插件 API 不兼容，混用代价极大。

**切换预案**：阶段一验证时，如 Tauri v2 实现不如预期（Rust 编译慢、前端通信复杂等），可切换到 Electron。前端 Vue 代码可复用，只换后端壳子。

### 1.2 模型通信

| 组件 | 方式 | 地址 |
|------|------|------|
| 审美评估模型 | HTTP REST API | Ollama localhost:11434 |
| 图片传递 | base64 编码 | 嵌入 API 请求体 |
| 输出解析 | 结构化 JSON | prompt 中要求模型输出 JSON 格式 |

#### Ollama 调用的并发模型

> **现实约束**：Ollama 默认对**同一模型**串行处理推理请求；GPU 显存限制了多模型并发的可行性（InternVL3.5-8B 约 14GB，gemma3:4b 约 4GB，消费级 GPU 跑不动两个 8B 同时推理）。"客户端开 5 路并发"无法绕过这两个限制。

实际并发策略：

| 维度 | 策略 |
|------|------|
| 同模型串行 | 同一时刻只有一个 InternVL3.5 请求在 GPU 上跑，多余请求在 Ollama 队列等待 |
| 跨模型分时 | Spike 1 已取消默认轻量预判层，默认不再进行 gemma4:e4b → InternVL3.5 的阶段切换。gemma4:e4b 仅作为显存不足时的降级模型 |
| 客户端请求队列 | 工具端维护 FIFO 队列（深度可配，默认 3），目的是限制 UI/IPC 并发和避免阻塞主线程，不是真正 GPU 并发 |
| 实际并发数 | InternVL3.5 同模型串行；缩略图生成等 CPU 任务用 Tauri async command + `spawn_blocking` 放到后台线程 |

> **实测决策（2026-06-16 Spike 1/2）**：gemma4:e4b 对 63 张樱花同场景照片 100% 输出 keep，平均 6.58s/张，区分力和速度均不达标；默认改为**全量 InternVL3.5 单层架构**。InternVL3.5 稳定性测试 30 张 × 3 次，结论一致率 100%，平均 2.76s/次。

#### Prompt 版本管理

评估缓存 key 包含 `prompt_version`，因此 prompt 文件必须有可靠的版本标识。规则：

1. 每个 prompt YAML 首部必须包含：
   ```yaml
   version: 3              # 语义版本号，手动维护
   content_hash: <auto>    # 文件主体的 SHA-256，工具自动校验
   ```
2. 工具启动时校验 `content_hash` 与文件实际内容是否一致；不一致则提示"prompt 已修改但版本号未升，请确认后再继续"，阻塞进入评估流程
3. 缓存 key 使用 `version` 字段而非 hash——这样允许"语义不变的格式调整"不打破缓存，但**任何会改变模型输出语义的改动都必须升 version**
4. 调优期可用 `--no-cache` 命令行参数旁路缓存，避免"prompt 改了缓存命中旧结果"的迷惑性 bug

### 1.3 数据存储

| 数据 | 存储方式 | 路径/说明 |
|------|---------|----------|
| 照片分类状态 | 文件夹位置 | 照片在 保留/删除/待定/ 哪个文件夹就是状态 |
| 审美档案 + 技术档案 | XMP sidecar | 跟照片走，Lightroom 可读 |
| 用户偏好 | 本地 JSON 文件 | 工具配置目录下 (~/.snappick/preferences.json) |
| 决策历史 | 本地 SQLite | 学习机制的结构化数据 |
| Prompt 模板 | YAML 配置文件 | prompts/ 目录下，按场景分类 |
| 会话进度 | .session 文件 | 工作目录下，断点续传用 |
| 应用配置 | 本地 JSON 文件 | 工具配置目录下 (~/.snappick/config.json) |

### 3.2 区域布局原则

> **核心修正**：摄影筛选的真实比例是 **保留 ≪ 删除**（典型 1:5 ~ 1:10），把"保留区"放最大不符合实际操作流。主区域应当承载**当前正在处理的照片**，分类结果区只需要展示计数和最近移入项即可。

| 区域 | 位置 | 大小 | 内容 |
|------|------|------|------|
| **待处理区** | 上方主区域 | ~70% 面积 | 尚未做出最终决定的照片，按系统建议排序（淘汰候选优先 → 推荐保留 → 待审查）。卡片右下角显示系统建议徽标（✓建议保留 / ✗建议淘汰 / ?不确定）|
| **保留区** | 下方左 1/3 | 紧凑卡片 | 已确认保留的照片，按时间倒序，最近移入项在前 |
| **待定区** | 下方中 1/3 | 紧凑卡片 | 用户主动标记"待定"的照片，收尾阶段需清零 |
| **删除区** | 下方右 1/3 | 紧凑卡片 | 已确认删除的照片，按时间倒序，方便撤销误删 |

- 待处理区是**焦点队列**，决策动作（P/X/U）从此区移出到对应分类区
- 下方三区是**结果展示**，主要用于查看计数、回看最近决策、撤销误操作
- 操作动画：照片从上方"飞"到下方对应区域，强化"决策完成"的反馈感

### 3.3 初始状态与分类区显示规则

- 启动后：800 张全部进入待处理区，按系统建议排序
- 下方分类区在断点续传场景下加载已有分类的照片
- 分类区显示数量上限（每区显示最近 50 张），超出部分点击区域标题展开滚动列表
- 分类区双击照片可"召回"到待处理区重新决策（等同于 Ctrl+Z 单步撤销）

### 3.3 照片卡片

保留区卡片较大(160px宽)，待定区和删除区卡片较小(100px宽)。

每张卡片显示：
- 缩略图
- 左上角状态标记（✓保留 / ?待定 / ✗删除）
- 右下角综合评分

选中态：红色边框 + 脉冲动画

### 3.4 详情面板

右侧固定 320px 宽，不包含图片预览区域。用户如需查看大图，**双击照片卡片调用系统默认图片查看器打开**。

详情面板内容从上到下：
1. 文件信息（缩略图 + 文件名 + EXIF 摘要）
2. 审美档案（核心亮点 + 四维度分数条）
3. 技术档案（各维度分数 + 修复建议）
4. 综合建议 + 操作按钮（P/X/U）
5. 标签

### 3.5 查看大图

**双击照片卡片 → 调用操作系统默认图片查看器打开**

- Windows: 调用 `start` 命令或 Shell API 打开
- 不在工具内实现图片放大功能，减少复杂度
- Z 键放大功能取消，用系统查看器替代

## 四、弹框交互设计

### 4.1 理由选择弹框

#### 触发逻辑

> **删除时弹理由（推翻系统 = 高价值信号）；保留时按抽样规则弹理由（捕获偏好画像，详见设计文档 8.5）。**

| 场景 | 是否弹理由 | 原因 |
|------|-----------|------|
| 系统建议保留 → 你保留（命中） | 抽样弹（冷启动 1/3，校准 1/10，信任 1/30） | 偏好画像核心数据来源 |
| 系统建议保留 → 你保留（新场景前 5 次） | ✅ 必弹 | 新场景需要快速积累偏好 |
| 系统建议保留 → 你删除 | ✅ 必弹删除理由 | 你否定了系统——高价值学习信号 |
| 系统建议淘汰 → 你删除 | ❌ 不弹 | 一致，删除理由信号价值低（同意"差"无新信息）|
| 系统建议淘汰 → 你保留 | ✅ 必弹保留理由 | 你否定了系统——高价值学习信号 |
| 系统不确定 → 你保留 | ✅ 必弹保留理由 | 系统没判断，你提供了信号 |
| 系统不确定 → 你删除 | ✅ 必弹删除理由 | 系统没判断，你提供了信号 |

#### 弹框结构

```
┌─────────────────────────────────────────┐
│  选择原因                                    ✕ │
│                                           │
│  🧠 当前为学习阶段，建议选择原因帮助我更懂  │
│  你的偏好。随着学习深入，此弹框出现频次会    │
│  逐渐降低。                                │
│                                           │
│  ○ 没有亮点，平平无奇                      │
│  ○ 背景杂乱，画面不干净                    │
│  ○ 表情/姿态不好                          │
│  ○ 构图不舒服                              │
│  ○ 光线/氛围不好看                        │
│  ○ 有更好的同类照片了                      │
│  ○ 其他                                   │
│    ┌─────────────────────────────────┐    │
│    │ 请输入具体原因... (必填)         │    │
│    └─────────────────────────────────┘    │
│                                           │
│           [确认]                           │
└─────────────────────────────────────────┘
```

#### 删除理由选项

| # | 选项 | 说明 |
|---|------|------|
| 1 | 没有亮点，平平无奇 | 无任何突出之处 |
| 2 | 背景杂乱，画面不干净 | 环境干扰主体 |
| 3 | 表情/姿态不好 | 人像场景常见 |
| 4 | 构图不舒服 | 画面布局问题 |
| 5 | 光线/氛围不好看 | 光影或色调不理想 |
| 6 | 有更好的同类照片了 | 相对比较后淘汰 |
| 7 | 其他（需填写） | 选择后弹出必填输入框 |

#### 保留理由选项

| # | 选项 | 说明 |
|---|------|------|
| 1 | 画面好看，光线美 | 视觉吸引力 |
| 2 | 表情/情绪好 | 情感共鸣 |
| 3 | 瞬间难得，错过不再有 | 稀缺性 |
| 4 | 有故事感 | 叙事性 |
| 5 | 整体感觉对 | 直觉判断 |
| 6 | 后期空间大 | 技术可修性 |
| 7 | 其他（需填写） | 选择后弹出必填输入框 |

#### 交互规则

- **单选**：最重要的那个理由最有学习价值
- **选项 1-6**：选中后可直接确认
- **选项 7"其他"**：选择后出现必填输入框，不填无法确认
- **可跳过**：右上角 × 关闭弹框，直接完成操作（跳过不影响操作本身）
- **确认后**：照片移入对应区域，理由记录到决策历史

#### 渐退机制

| 阶段 | 条件 | 行为 |
|------|------|------|
| 冷启动期 | 前 100-200 次操作 | 不一致时必弹；一致时按 1/3 抽样 |
| 校准期 | 维度权重稳定 + 预测准确率 > 70% | 不一致时仍弹；一致时按 1/10 抽样 |
| 信任期 | 各场景 30+ 样本 + 高置信度 | 不一致时仍弹（罕见）；一致时按 1/30 抽样 |

## 六、虚拟滚动与缩略图加载方案

800+ 张照片的展示性能关键点：

- 使用虚拟滚动库（如 vue-virtual-scroller），只渲染可视区域的卡片
- 缩略图按需加载：滚动到可视区域时才生成/加载缩略图
- 缩略图尺寸：保留区 160px 宽，待定/删除区 100px 宽
- 缩略图缓存：Rust 后端生成后存入临时目录，避免重复生成

### 6.1 Spike 实测结论

> **2026-06-16 Spike 3 实测**：
> 1. 一次性返回全部缩略图 base64 会导致 WebView 崩溃；
> 2. 同步 Tauri command 中解码图片会冻结 UI；
> 3. `convertFileSrc` 在当前 Windows + Tauri v2 配置下缩略图未展示，需后续优化；
> 4. 当前可用方案为：**逐张按需 base64 + 并发 3 路 + Rust `spawn_blocking` 后台解码**。

### 6.2 v1 缩略图加载规则

| 规则 | 说明 |
|------|------|
| 禁止批量返回图片数据 | IPC 每次只返回单张缩略图，不能一次返回 100/800 张 |
| 并发限制 | 前端最多同时请求 3 张缩略图，超出排队 |
| 后台线程 | Rust 端图片解码 / 缩放 / JPEG 编码必须放入 `spawn_blocking` |
| 缓存优先 | 缩略图首次生成后写入系统 cache 目录，后续命中缓存 |
| 后续优化方向 | 尝试修复 `convertFileSrc` / asset protocol，使 `<img>` 直接读缓存文件，避免 base64 字符串驻留内存 |

## 八、文件移动的实现约束

> **核心原则：状态由文件位置决定，所以文件移动必须可靠到等同事务。**

### 8.1 配对原子化

JPG + RAW + XMP 视为一个事务单元。Rust 后端实现时：

```
1. 收集本次操作涉及的所有文件路径（同名不同后缀）
2. 预检查：源全部可读 + 未被占用 + 目标可写 + 目标无同名冲突
3. 顺序移动；任一失败 → 已成功的回滚到原位 → 整体报错
4. 全部成功 → 落入撤销栈
```

### 8.2 跨磁盘检测

启动时通过对比工作目录与系统盘的卷标判断是否跨盘。跨盘场景（NAS、移动硬盘）move 会退化为复制+删除，秒级承诺失效。首次启动弹窗需显式提示，让用户知情。

### 8.3 撤销栈一致性

撤销栈仅记录**已成功完成**的移动。撤销时若回滚失败（目标文件被外部占用、磁盘问题等），从栈中弹出该项并提示用户具体哪一步失败，绝不允许"假装撤销成功"导致内存状态与磁盘状态错位。

栈深度 50 步，超出后丢弃最早的。

### 8.3.1 撤销栈的会话边界

> **明确规则**：撤销栈**仅在当前会话有效**，关闭软件后清空。这是工程权衡：跨会话撤销栈需要持久化操作历史 + 处理"撤销栈中的目标文件已被外部移动/删除"的复杂边界，收益不抵成本。

为弥补"关软件后撤不回"的安全感缺失，依靠两层兜底：

1. **删除 ≠ 永久消失**：删除子文件夹的内容只有用户主动清空才真正删除。需要恢复某张照片，从删除文件夹手动拖回或工具内"召回"即可
2. **首次启动弹窗显式告知**：在欢迎弹窗（[ArchitectureUI §4.2](#42-首次启动确认弹框)）增加一行说明——"撤销（Ctrl+Z）仅在本次会话有效；关闭后请从删除文件夹手动恢复"

如果未来用户反馈跨会话撤销是刚需，再单独评估持久化撤销栈的方案——但不在 v1 路线内。

### 8.4 启动自检与幽灵态处理

每次启动时：

1. 扫描三个子文件夹的实际文件清单
2. 与 `.session` 文件中上次会话的清单对比
3. 发现差异（手动移动 / 删除 / 新增）→ 不静默处理，进入"需用户确认"流程
4. 用户确认后才更新 `.session` 与 SQLite 中的决策记录

### 8.5 评估缓存与文件身份

照片的评估结果缓存 key 必须与文件**身份**绑定，而非路径——文件在保留/待定/删除子文件夹间移动时，评估结果不应失效。建议使用文件内容哈希（首次评估时计算，存入 SQLite + XMP）作为身份标识。
