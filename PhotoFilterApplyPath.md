## Spike 阶段：风险验证（4-6h）

> **开工前必做。三个不确定性如果不验证就直接进阶段零，后面的架构决策都是空中楼阁。**

### Spike 1：模型分流可行性（已完成，2026-06-16）

- 实测素材：63 张樱花同场景照片
- 实测模型：gemma4:e4b
- 结果：100% keep，0% drop/unsure，平均 6.58s/张
- **决策结论**：轻量预判层不可行，取消两层架构，默认全量走 InternVL3.5

### Spike 2：评分稳定性测试（已完成，2026-06-16）

- 实测素材：30 张照片 × 3 次 = 90 次调用
- 实测模型：InternVL3.5（bInternlVL35:8b）
- 结果：结论一致率 100%，分数抖动中位数 0，JSON 解析成功率 100%，平均 2.76s/次
- **决策结论**：评分稳定性达标；但分数集中在 85/90/95，需保留 5 档展示并后续调优 prompt 区分度

### Spike 3：缩略图加载与 Tauri IPC 性能（1-2h）

- 800 张真实照片
- 用 Tauri 后端生成缩略图 + IPC 传到 Vue 前端 + 虚拟滚动展示
- 测量首屏时间、滚动 FPS、内存占用
- **决策点**：滚动卡顿（< 30 FPS）或首屏 > 5s 则必须先优化（缓存策略 / 缩略图尺寸 / IPC 协议），不要带病进阶段一

### 产出

`PhotoFilterSpike.md`——记录三项实测数据和最终决策。这份文档会修正本路线规划里的若干假设。

## 阶段一：手动快筛（12-16h）

### 产出

不接任何 AI，纯键盘操作快速筛选。操作即文件移动，比文件管理器快得多。

这是工具的**底线价值**——即使没有 AI，工具也已经可用。

### 任务

| 任务 | 估时 | 说明 |
|------|------|------|
| 子文件夹创建 + 首次启动确认弹窗 | 0.5h | 创建 保留/删除/待定 子文件夹 |
| 三区域布局 + 虚拟滚动 + 卡片选中态 | 4-6h | Vue 主框架 + vue-virtual-scroller |
| 键盘操作（P/X/U/星标/方向键/Tab） | 1h | 全局事件 + 焦点态 |
| 文件移动事务化（含原子性、回滚、跨盘检测） | 3-5h | Rust 后端，详见架构文档 §8 |
| 单张预览 + 双击调用系统查看器 | 0.5h | 临时放大用系统查看器替代 |
| Ctrl+Z 撤销栈（与文件状态强一致） | 2-3h | 撤销栈深度 50，回滚失败处理 |
| 断点续传 + 启动自检 | 1h | 扫描子文件夹 + 与 .session 对比 |

### 验证点

- 用真实照片完整走一遍筛选流程，操作是否流畅
- 文件移动是否秒级完成
- 配对移动是否可靠（不会出现留了 JPG 留了 RAW）
- **故意中断**：移动期间杀进程、拔 U 盘、占用文件——是否能正确恢复

### 快捷键

| 动作 | 快捷键 |
|------|--------|
| 保留 | P |
| 删除 | X |
| 待定 | U |
| 星标 1-5 | 1 2 3 4 5 |
| 去除星标 | 0 |
| 下一张 | → / 空格 |
| 上一张 | ← |
| 下一组 | Tab |
| 放大查看 | Z / F |

## 阶段二：技术初筛（1h）

### 产出

传统算法做硬过滤和技术评估，不需要 AI 模型。硬伤照片自动淘汰，技术评分辅助判断。

### 任务

| 任务 | 说明 |
|------|------|
| 硬过滤 | 模糊检测(Laplacian)、纯黑/纯白帧、文件损坏 |
| 基础技术评估 | 曝光分析(直方图)、清晰度评估、EXIF 提取 |
| XMP 读写 | 写入评分和标签，兼容 Lightroom/Capture One |
| 评估结果展示 | 技术档案 + 按技术分排序 |

### 验证点

- 硬过滤误杀率——拿 100 张真实照片跑一遍，好片是否被误判
- 技术评估和直觉是否对得上

## 阶段三：AI 审美评估（8-12h）

### 产出

工具从"快筛器"升级为"智能助手"——审美驱动留存价值判断。

**这是核心差异化能力。**

### 3a：Prompt 验证（4-6h）

用 Dify 作为 prompt 实验室，验证后再迁移到工具。

| 任务 | 说明 |
|------|------|
| Dify 部署 + 连接 Ollama | Docker 本地部署，连 InternVL3.5（gemma4:e4b 仅作为降级测试） |
| 场景识别 prompt 调通 | 人像/风景/街拍/活动/建筑 分类 |
| 审美评估 prompt 调通 | **核心耗时项**——反复迭代，找合适的 prompt |
| 深度评估 prompt 调通 | 边界照片的细致评估 |
| 组内对比 prompt 调通 | 两张照片的对比判断 |

**产出：验证过的 prompt 模板 + 模型路由逻辑**

> **迁移风险预警**：Dify 与工具端 Rust 实现的 prompt 模板语法、变量替换、输出 schema 验证、自动重试机制可能不一致。Dify 上跑得好的 prompt 直接搬到工具端**效果可能下降**——这不是模型问题，是工具链问题。3b 末尾必须做迁移一致性验证。

### 3b：集成到工具（4-6h）

| 任务 | 估时 | 说明 |
|------|------|------|
| Ollama API 调用封装 + 请求队列 | 1h | HTTP + base64 + 队列深度 8（详见架构 §1.2）|
| Prompt 模板配置系统（YAML + content_hash 校验） | 0.5h | 按场景/模式选择，启动时校验 hash |
| 评分稳定性约束实现 | 1h | temp=0、固定 seed、整数化、5 档展示、self-consistency |
| 评估结果缓存（SQLite，key=哈希+模型版本+prompt 版本+语境） | 1h | 默认不重跑，支持 `--no-cache` 旁路 |
| InternVL3.5 单层评估流程 | 1h | Spike 1 已取消轻量预判层，全量直接评估 |
| 审美档案 UI 展示 | 1h | 四维度分数、核心亮点 |
| 评判语境推断 | 0.5h | RAW→后期意图、场景→评估标准调整 |

#### 3b 验收：Dify → 工具端迁移一致性

**必做**：取 3a 阶段在 Dify 上验证好的同一组 30 张照片，在工具端 Rust 实现下用同样的 prompt + 模型 + 参数复跑。

| 指标 | 阈值 | 处理 |
|------|------|------|
| 结论一致率（保留 / 淘汰 / 不确定） | ≥ 90% | 通过，进入下一阶段 |
| 分数偏差 | 平均 ≤ ±5 分 | 通过 |
| JSON 解析成功率 | 100% | 否则补 schema 修复逻辑 |
| 任一指标不达标 | — | 排查工具链层差异（变量替换、温度参数、stop tokens、超时配置），修复后重测 |

**不要跳过这一步**——3a 调好的 prompt 一旦迁移失败，整个阶段三的 prompt 调优时间就白费了。

### 验证点

- 100 张真实照片：AI 判断和你的直觉一致率多少？目标 ≥ 70%
- 800 张全量跑一遍，速度是否可接受？
- "痛苦抉择"照片（高审美 + 致命技术伤）是否正确呈现？
- **结论稳定性**（与 Spike 2 同方法重测）≥ 95%
- **Dify → 工具端迁移一致率** ≥ 90%

## 阶段四：智能分组（1h）

### 产出

相似照片自动分组，每组推荐关键帧，一键选优。

### 任务

| 任务 | 说明 |
|------|------|
| 相似度计算 + 自动分组 | 时间戳邻近 + pHash 距离阈值聚类 |
| 三层差异检测（EXIF → pHash/SSIM → 局部特征） | 本地算法，不调模型；详见设计文档 §11.2 |
| 关键帧推荐 | 基于审美评分选组内最优 |
| 组内成员分数继承 | 差异极小者继承关键帧分数；差异显著者单独评估 |
| 组内批量操作 UI | "就选推荐帧，其余删除" |
| 包围曝光识别 | EXIF 包围标记检测 → 素材组 |

### 验证点

- 连拍组是否正确识别
- 推荐帧你是否认同

## 阶段五：学习个性化（1h 实现，长期隐性能力）

### 产出

工具越用越懂你。开发实现 1h，但学习效果是长期积累的。

### 任务

| 任务 | 说明 |
|------|------|
| 理由选择 UI + 渐退机制 | 删除时弹理由选项 + 保留时按抽样规则弹，按置信度消退 |
| 决策记录 + 偏差分析 | 存储每次决策，提取偏好模式 |
| 偏好画像展示 | 你的审美偏好摘要 |
| 偏好应用到评估 | 学到的偏好影响后续评分 |
| 遗忘 / 重置 / 风格快照 | 偏好衰减和管理 |
| **偏好导出 / 导入**（JSON） | 跨设备迁移、风格快照分享，仅偏好元数据不含照片本身 |

### 长期隐性能力

学习机制不是"做完就结束"的功能，而是一个**持续运行、持续生效**的隐性能力：

- 每次使用都在积累偏好数据
- 偏好随使用量增加而收敛（10次→低置信度，30次→中，100次→高）
- 人的审美会变——遗忘机制让系统跟着你变
- 长远可训练个人化轻量模型（当数据积累足够时）

## 时间分布的真实情况

```
Spike 验证:    ~5h    (开工前必做，决定后续架构)
纯代码开发:    ~36-50h (vibe coding 提效后的真实区间)
prompt 调优:   不确定  (阶段三 3a 的核心耗时，取决于模型能力和迭代次数)
长期学习积累:   持续    (阶段五上线后每次使用都在积累)
```

**真正的瓶颈不在写代码，在于：**

1. **Prompt 质量** — 决定审美评估的方向是否对。阶段三 3a 的 4-6h 是预估，实际可能更多
2. **学习数据积累** — 偏好模型需要真实决策数据喂养，这不是开发时间能加速的，是用出来的
3. **Spike 验证质量** — 三个验证点任何一个通过得潦草，后面架构都会还债

## 风险与缓解

| 风险 | 阶段 | 缓解 |
|------|------|------|
| 本地模型审美判断不准 | 三 | Dify 验证阶段暴露问题；prompt 调不好换模型或加云端增强 |
| 800 张照片评估太慢 | 三 | 分级评估 + 工具端并发控制 |
| 缩略图加载卡顿 | 零/一 | 虚拟滚动 + 懒加载 |
| Ollama 服务不稳定 | 三 | 重试机制 + 降级提示（标记为"AI 不可用"） |
| Prompt 调优耗时超预期 | 三 3a | 先用简单 prompt 上线，后续持续迭代优化 |
